## 一、本次会话完成的任务

### 1.1 优化 d2 图生成 —— 双向边 `<->` 支持

**问题描述**：Mocker 的 edge 可以包含多个 flow statement，数据可能沿两个方向流动（如 `<add_str>` 的 hello.h→world.words 和 world.new→hello.out_str）。但 d2 图只画了 `hello -> world` 单向。

**根因**：
- 原代码遍历 `pkg.Topology.Edges`（slice），但 package-level edge 可能在 Topology.Edges 里没列
- 去重逻辑只按 `src -> dst` 方向去重，没有检测反向流

**修改的文件**：

| 文件 | 改动概要 |
|------|----------|
| `circle/internal/d2gen/d2gen.go` | 重构为 4-phase 架构：预扫描双向 top-level edge → 收集所有边 → 检测 SubEdge 双向对 → 去重输出。使用 `<->` 箭头替代两个单箭头 |
| `circle/internal/d2gen/d2gen_test.go` | 新增 7 个测试：双向边检测、标签、SubEdge 双向对、单向边回归 |
| `example/debug_test/05-graph.d2` | 更新示例输出 |

**效果**：
```d2
hello <-> world: "<add_str>"   # 双向（之前 hello -> world 单向）
hello -> Println: ">>"          # SubFlow 单向
Println -> write: ">>"          # SubFlow 单向
```

**关键 bug 修复**：
- 之前迭代 `pkg.Topology.Edges`（slice）漏掉了 package-level edge
- 改为迭代 `pkg.Edges`（map），确保所有 edge 都被处理

### 1.3 隐式 SubEdge 源推断 —— `<add_str> w` 语法糖

**问题**：之前必须写 `h <add_str> w`（显式指定源变量），用户希望可以写 `<add_str> w` 由编译器推断。

**方案**：根据 edge body 的第一个 FlowStmt 的 callee 入 port 类型，在 caller 节点 body 中找**类型唯一匹配**的非 input 变量作为源。

**修改的文件**：

| 文件 | 改动概要 |
|------|----------|
| `circle/internal/parser/parse_decl.go` | 新增 `isImplicitSubEdgeSig` + `parseImplicitSubEdgeDecl`：识别 `<add_str> w` 模式，生成 `SubEdgeDecl{Src:""}` |
| `circle/internal/ir/lower.go` | 新增 `inferSubEdgeSrc` 函数：按 edge body + AST 类型匹配推断源；新增 `semTypeToIRType` / `lookupCalleeInputType` |
| `circle/internal/ir/lower_infer_test.go` | 新增 4 个推断测试 |
| `circle/internal/codegen/emit.go` | 新增 `__implicit__` 标记检查，推断失败时输出清晰错误诊断 |

**推断规则**（`ir/lower.go:inferSubEdgeSrc`）：
1. 从 top-level edge body 的第一个 FlowStmt 拿到 callee 的入 port（如 `world.words`）
2. 查 callee node 拿到该入 port 的类型（如 `str`）
3. 扫描 caller 节点的 AST 中"非 input"变量集（VarDecl + FieldDecl + FlowDecl）
4. 找**类型唯一匹配**的变量
5. 0 个或 ≥2 个候选 → 推断失败，标记 `__implicit__`

**错误诊断**（推断失败时）：
```go
// ERROR: implicit SubEdge <add_str> w could not infer source
// HINT: change '<add_str> w' to 'X <add_str> w' (specify source explicitly)
```

## 五、核心设计理念

### 5.1 控制流透明传递

Mocker 的 for/while/if 语法与 Go 基本一致，因此 IR 层将控制流语句转换为文本字符串（`IRSimpleStmt.Text`），codegen 直接 emit 该文本。

**理由**：
- **语法兼容性高**：C 风格 `for(i:=0; i<3; i++)` 和 Go 的 for 语法几乎一致
- **实现简洁**：无需在 IR 层解析控制流的内部结构
- **代价**：无法在 IR 阶段做 for 循环内的变量类型推断

### 5.2 复合赋值的解糖

`x += y` 被解析为 `AssignStmt{Lhs:[x], Rhs:y, Compound:"+="}`，codegen 直接 emit 成 Go 的 `x += y`。

`i++` 被解析为 `AssignStmt{Lhs:[i], Compound:"+", Rhs:nil}`（Rhs=nil 标记为自增），codegen 中 emit 为 `i++`。

### 5.3 隐式 SubEdge 推断

推断基于 **类型唯一匹配** 原则：edge body 的第一个 FlowStmt 的 last step 声明了 callee 的入 port 类型，caller 节点 body 中必须有且仅有一个 `非 input` 变量匹配该类型。

### 5.4 双向边检测

d2gen 使用 4-phase 架构：
1. **Phase 1**：预扫描所有 top-level edge 的 Flow，检测是否有反向流（`op.Src == ek.Dst && op.Dst == ek.Src`）
2. **Phase 2**：收集所有边（main_entry / SubEdge / SubFlow / top-level），双向 top-level edge 覆盖 SubEdge
3. **Phase 3**：检测 SubEdge/SubFlow 双向对（A→B 和 B→A 都存在）
4. **Phase 4**：去重并输出，双向用 `<->`，单向用 `->`
