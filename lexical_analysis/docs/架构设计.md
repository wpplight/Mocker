# glex 架构设计

> **glex** = **G**o **L**ex**er Generator**
> 类比 gRPC：从声明式定义文件（`.glex`）自动生成 Go 词法分析器包。

---

## 一、项目定位

| 角色 | 类比 |
| --- | --- |
| `protoc` 读 `.proto` → 生成多语言桩代码 | **glex** 读 `.glex` → 生成 Go 词法分析器包 |
| 用户写 schema，工具产出代码 | 用户写 token 定义，工具产出可 import 的包 |
| 产物独立编译、独立运行 | 生成的 lexer 包**就是普通 Go 包** |

### 用户视角的 3 步工作流

```bash
# 1. 写 token 定义（用 _ 分割 type 和 name）
$ cat tokens.glex
package: mylexer
tokens:
  ID:        '[a-zA-Z_][a-zA-Z0-9_]*'   # type=ID, name=ID
  NUM:       '[+\-]?[0-9]+'             # type=NUM, name=NUM
  REAL:      '[+\-]?[0-9]+\.[0-9]+'     # type=REAL, name=REAL
  KW_IF:     'if'                       # type=KW, name=IF
  KW_ELSE:   'else'                     # type=KW, name=ELSE
  OP_ADD:    '\+'                       # type=OP, name=ADD
  OP_SUB:    '-'                        # type=OP, name=SUB
  OP_LT:     '<'                        # type=OP, name=LT
  OP_EQ:     '=='                       # type=OP, name=EQ
  SEP_SEMI:  ';'                        # type=SEP, name=SEMI
  ...

# 2. 跑 glex 生成
$ glex -i tokens.glex -o ./mylexer --with-tests
✓ 解析 21 个 token 定义
✓ 构建 NFA（25 状态）
✓ 转换为 DFA（19 状态）
✓ 最小化 DFA（14 状态）
✓ 生成 ./mylexer/{token,dfa,scan,lexer,lexer_test}.go

# 3. 在自己的程序里用
$ cat main.go
import "myproject/mylexer"
for _, t := range mylexer.Tokenize(src) {
    fmt.Println(t.Type, t.Type.Kind(), t.Lexeme)
}
// 输出：T_ID ID "x" / T_OP_ADD OP "+" / T_KW_IF KW "if"
```

## 四、glex 内部模块详解

### 4.1 DSL 解析（internal/dsl）

**职责**：把 `tokens.glex`（YAML）解析为内存中的 `TokenSpec` 列表。

**数据结构**：
```
TokenSpec {
    Key   string   // 原始 DSL key，如 "OP_ADD" / "ID"
    Type  string   // 解析出的 type，下划线前的部分
    Name  string   // 解析出的 name，下划线后的部分
                    // （无下划线时 Type = Name = Key）
    Regex string   // 原始正则字符串
}

SpecFile {
    Package string       // 输出包名
    Tokens  []TokenSpec  // token 定义
    Order   []string     // 保留 DSL 中的定义顺序（用于冲突解决）
}
```

**命名约定（_ 分割 type 和 name）**：

| DSL key | Type | Name | 含义 |
| --- | --- | --- | --- |
| `ID` | `ID` | `ID` | 这种 type 只有一种 token，name == type |
| `NUM` | `NUM` | `NUM` | 同上 |
| `REAL` | `REAL` | `REAL` | 同上 |
| `OP_ADD` | `OP` | `ADD` | OP 类里的 `+` 运算符 |
| `OP_SUB` | `OP` | `SUB` | OP 类里的 `-` 运算符 |
| `KW_IF` | `KW` | `IF` | KW 类里的 `if` 关键字 |
| `SEP_SEMI` | `SEP` | `SEMI` | SEP 类里的 `;` 界符 |

> - **Type**（"这是哪一类 token"）：用于分组和冲突解决
> - **Name**（"具体是哪一个"）：用于精确识别

**输入示例**（`examples/tokens.glex`）：
```yaml
package: mylexer
tokens:
  # 单实例 type（无下划线）：name == type
  ID:           '[a-zA-Z_][a-zA-Z0-9_]*'
  NUM:          '[+\-]?[0-9]+'
  REAL:         '[+\-]?[0-9]+\.[0-9]+'

  # 关键字（KW 类）
  KW_IF:        'if'
  KW_ELSE:      'else'
  KW_FOR:       'for'
  KW_WHILE:     'while'

  # 运算符（OP 类）
  OP_ADD:       '\+'
  OP_SUB:       '-'
  OP_MUL:       '\*'
  OP_DIV:       '/'
  OP_ASSIGN:    '='
  OP_EQ:        '=='
  OP_LT:        '<'
  OP_GT:        '>'
  OP_LE:        '<='
  OP_GE:        '>='

  # 定界符（SEP 类）
  SEP_LPAREN:   '\('
  SEP_RPAREN:   '\)'
  SEP_LBRACE:   '\{'
  SEP_RBRACE:   '\}'
  SEP_SEMI:     ';'
  SEP_COMMA:    ','
```

**解析逻辑**（伪代码）：
```
for key, regex in yaml.tokens:
    parts = split(key, "_", max=1)  # 最多切一刀
    if len(parts) == 1:
        type, name = parts[0], parts[0]   # "ID" → type=ID, name=ID
    else:
        type, name = parts[0], parts[1]   # "OP_ADD" → type=OP, name=ADD
    spec = TokenSpec{Key:key, Type:type, Name:name, Regex:regex}
    tokens.append(spec)
    order.append(key)   # 记住定义顺序（冲突解决用）
```

### 4.2 正则 AST & 解析（internal/regex）

**支持的语法子集**：

| 语法 | 含义 | 示例 |
| --- | --- | --- |
| `c` | 字面字符 | `a` `;` |
| `\x` | 转义 | `\+` `\\` |
| `[abc]` `[a-z]` | 字符类 | `[0-9]` `[a-zA-Z_]` |
| `[^abc]` | 取反字符类 | `[^ \t\n]` |
| `a b` | 连接 | `ab` |
| `a\|b` | 选择 | `if\|else` |
| `a*` | Kleene 星 | `[a-z]*` |
| `a+` | 一次或多次 | `[0-9]+` |
| `a?` | 零次或一次 | `[\-+]?` |
| `(a)` | 分组 | `(if\|else)` |
| `.` | 任意单字符 | `.*` |

**AST 节点**：
```
interface Regex
├── Literal(ch byte)
├── CharClass(chars []byte, negate bool)
├── Concat(left, right Regex)
├── Union(left, right Regex)
├── Star(inner Regex)
├── Plus(inner Regex)
├── Optional(inner Regex)
└── Group(inner Regex)   // 透明分组
```

**不支持**：锚点 `^`/`$`、反向引用、lookahead/lookbehind（DFA 整体匹配，不需要）。

### 4.3 NFA 构造（internal/nfa）

**Thompson 构造法**：把每个 AST 节点转成一个 ε-NFA 片段（start, end）。

| AST 节点 | NFA 构造 |
| --- | --- |
| `Literal('a')` | `→ [a] → ( )` |
| `CharClass({a,b})` | `→ [a,b] → ( )` |
| `Concat(a, b)` | a 的 end → ε → b 的 start |
| `Union(a, b)` | 新 start →ε→ a；新 start →ε→ b；a.end, b.end →ε→ 新 end |
| `Star(a)` | 新 start →ε→ 新 end；新 start →ε→ a.start；a.end →ε→ 新 end；a.end →ε→ a.start |
| `Plus(a)` | a 的 NFA + a.end →ε→ a.start |
| `Optional(a)` | 新 start →ε→ 新 end；新 start →ε→ a.start；a.end →ε→ 新 end |

**合并 NFA**：新建超级起始状态 S0，从 S0 用 ε 边连到**每个 token NFA** 的起点；每个 token NFA 的终点带"接受 → 哪个 token"标签。

### 4.4 DFA 构造（internal/dfa/subset.go）

**子集构造法**（NFA → DFA）：
```
worklist = [ε-closure({S0})]
while worklist 不空:
    取出一个 DFA 状态 S (= NFA 状态集合)
    for 每个字符类 c:
        T = ε-closure(move(S, c))
        if T 非空:
            S --c--> T
            if T 不在 DFA 中: 加入 worklist
接受态 = 闭包内含 NFA 接受态的 DFA 状态
```

**字符类处理**：DFA 表的列是**字符类**而不是单个字符，扫描时用 `classify(byte) int` 把字符映射到类编号。

### 4.5 DFA 最小化（internal/dfa/minimize.go）

**划分等价类**（Hopcroft 算法简化版）：
1. 初始划分：`P = {接受态组, 非接受态组}`
2. 对每个组 `g`，对每个输入类 `c`，看组内状态经 `c` 转移到的"目标组"是否一致
3. 不一致则拆分
4. 重复直到 P 不变

**意义**：减少状态数 → 生成的表更小、生成的 lexer 更快。

### 4.6 代码生成（internal/codegen）

**核心**：把内存中的 DFA + TokenSpec 渲染为 Go 源文件。

| 生成的 .go | 内容来源 | 模板形式 |
| --- | --- | --- |
| `token.go` | TokenSpec 列表 | `enum TokenType { ... }` + `struct Token { ... }` + `Kind()` + `Name()` |
| `dfa.go` | DFA 表 + 字符类表 | `[NStates][NClasses]int{...}` + `classify` |
| `scan.go` | 通用 DFA 模拟器（**模板固定**） | 长匹配算法，**不依赖具体 token** |
| `lexer.go` | 通用入口（**模板固定**） | `Tokenize` 循环跳过空白 + 调 scan |
| `lexer_test.go` | TokenSpec 列表 | 每个 token 1 个基础测试 |

**`token.go` 生成细节**（由 DSL `OP_ADD` 等自动产出）：

```go
type TokenType int

const (
    T_ID TokenType = iota
    T_NUM
    T_REAL
    T_KW_IF
    T_KW_ELSE
    T_KW_FOR
    T_KW_WHILE
    T_OP_ADD
    T_OP_SUB
    ...
    T_SEP_SEMI
    T_SEP_COMMA
)

// Kind() 返回 token 所属的 type（"OP" / "SEP" / "KW" / "ID" 等）
// 同一 type 的 token 共享 Kind
func (t TokenType) Kind() string { ... }

// Name() 返回 token 的具体 name（"ADD" / "IF" / "ID"）
func (t TokenType) Name() string { ... }
```

**调用方用法**：
```go
// 精确匹配
if tok.Type == T_OP_ADD { ... }

// 分组匹配（任何 OP）
if tok.Type.Kind() == "OP" { ... }
```

**关键点**：`scan.go` 和 `lexer.go` 是**几乎所有项目通用的模板**，只是表数据变化。只有 `token.go` 和 `dfa.go` 真正按 token 集定制。

glex -i tokens.glex -o ./out -q

## 七、关键设计决策

| 决策 | 方案 | 理由 |
| --- | --- | --- |
| 工具名 | **glex**（Go Lexer Generator） | 简短、好记、类似 protoc |
| 输入格式 | **YAML**（可扩展到 JSON） | 易读、声明式 |
| 命名约定 | **`_` 分割 type 和 name** | 关键字和普通 token 统一表达，无须 `keywords` 字段 |
| 正则语法 | **标准子集** | 够用，不过度工程化 |
| NFA → DFA | **子集构造法** | 经典、教科书级 |
| DFA 最小化 | **划分等价类** | 减小表大小 |
| 代码生成 | **`text/template` + 字面量拼接** | 不引第三方模板引擎 |
| 字符类 | **按类分列** | 控制 DFA 表宽度 |
| 冲突解决 | **最长匹配 + 定义顺序优先** | 标准 lex 行为，行为可预测 |
| `+`/`-` 二义 | **DFA 设计 + 最长匹配自动消解** | 无需特殊代码 |
| 关键字 | **与普通 token 一视同仁**（`KW_IF: 'if'`） | 词法层不混入特殊升级逻辑 |
| 错误恢复 | **跳过 1 字符 + ERROR token** | 简单、保证不死循环 |
| 位置追踪 | **Token 带 Line/Col** | 给后续语法分析用 |
| Type/Name API | **生成 `Kind()` 和 `Name()` 方法** | 精确 + 分组两种匹配方式都支持 |
| 输出风格 | **4 个文件 + 可选测试** | 关注点分离，生成的包结构清晰 |
| 依赖 | **glex 工具本身**：yaml.v3；**生成代码**：零依赖 | 工具用 lib；产物纯净 |

## 八、与 lex/flex 的对比

| 特性 | lex/flex | glex |
| --- | --- | --- |
| 输入格式 | `.l`（自创 DSL） | `.glex`（YAML） |
| 输出语言 | C | Go |
| 嵌入方式 | `#include` / 链接 | `import` |
| 学习曲线 | 中（要学 `.l` 语法） | 低（YAML 是常识） |
| 字符类处理 | 自动 | 自动 |
| 错误恢复 | 用户在 action 里写 | 默认跳过 1 字符 |
| 输出可读性 | 中（生成 C 代码） | 中（生成 Go 代码） |
| 工具链 | C 编译器 | Go 工具链 |
| 增量编译 | 整文件重生成 | 整文件重生成（同） |
| 集成 | `flex lexer.l && gcc ...` | `glex -i ... -o ... && go build` |

**glex 的差异化卖点**：YAML 比 `.l` 易读；生成的是可独立 import 的 Go 包，不需要手写 `yylex`/`yyaction`。

## 九、扩展性

| 扩展项 | 改动范围 | 难度 |
| --- | --- | --- |
| 加新正则语法（`^`, `$`） | `internal/regex/parse.go` | ★★ |
| 支持字符串字面量 token | `internal/regex/` | ★★ |
| 输出 Python/Java lexer | 加 `internal/codegen/python/` | ★★★ |
| 增量编译（只重生成变化部分） | `internal/dsl/` 加缓存 + 比对 hash | ★★★ |
| 多文件输入（拆分 token 定义） | `internal/dsl/` | ★ |
| 自定义错误恢复策略 | CLI flag 透传到生成代码 | ★★ |
| 关键字在 DFA 阶段就识别 | DFA 设计要改，refine 简化 | ★★ |
| 在线预览 DFA（dot 格式） | `internal/dfa/` 加 dot exporter | ★ |

## 十、构建与运行

### 开发 glex 工具

```bash
cd lexical_analysis
go build -o bin/glex ./cmd/glex

# 跑示例
./bin/glex -i examples/tokens.glex -o /tmp/mylexer --with-tests -v

# 测生成的包
cd /tmp/mylexer && go test -v

# 测 glex 自身
go test ./internal/...
```

### 用户使用 glex

```bash
# 1. 安装 glex
go install github.com/yourname/lexical_analysis/cmd/glex@latest

# 2. 写 tokens.glex
$EDITOR tokens.glex

# 3. 生成
glex -i tokens.glex -o ./lexer

# 4. 在项目里用
#   go.mod: require myproject v0.0.0
#   main.go: import "myproject/lexer"
go mod tidy
go build
```

## 十一、风险与权衡

| 风险 | 应对 |
| --- | --- |
| 正则子集不够用 | 文档列明支持范围；按需扩展 |
| DFA 状态爆炸 | 字符归类 + 最小化；通常 < 50 状态 |
| 生成的代码不优雅 | 接受——生成的代码机器看，**不追求人维护** |
| 字符类归类不直观 | 默认归类方案 + 文档化 |
| glex 工具本身的 bug 影响所有产物 | 充分测试 internal/；CI |
| 用户写错 token 名字（如重复） | glex CLI 做语义检查，给清晰错误 |
| YAML 解析失败 | glex 提示文件位置 + 行号 |

## 十二、里程碑

| 阶段 | 内容 | 验收 |
| --- | --- | --- |
| M1 | `internal/dsl` 解析 | 能读 `tokens.glex` 打印 `TokenSpec` 列表 |
| M2 | `internal/regex` AST + 解析 | 能解析 `[a-z]+\|[0-9]+` |
| M3 | `internal/nfa` Thompson + 合并 | 能为单个 token 输出 NFA 状态图 |
| M4 | `internal/dfa` 子集构造 | 能输出 DFA 状态表 |
| M5 | `internal/dfa` 最小化 | 对比最小化前后状态数 |
| M6 | `internal/codegen` 4 个文件 | `glex -i ... -o ...` 产出可 `go build` 的包 |
| M7 | `cmd/glex` CLI | 支持 `-i/-o/-p/--with-tests/-v/-h` |
| M8 | 端到端测试 | glex 生成 → 生成的包编译 → 跑通示例 input.txt |
| M9 | 文档 & 例子 | `examples/tokens.glex` + `examples/input.txt` |

---

> **总结**：glex 是一个**声明式 → 自动代码生成**的工具。架构核心是 5 个内部包（dsl/regex/nfa/dfa/codegen）+ 1 个 CLI（cmd/glex），输入 YAML 描述，输出标准 Go 包。产物的使用者完全感知不到生成器的存在——就像 grpc 客户端感知不到 protoc 的存在一样。
